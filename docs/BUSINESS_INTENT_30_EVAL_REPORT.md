# 一期业务意图 30 Query 真实链路评测

## 1. 评测目的

本次评测验证一期 Demo 的业务意图是否能在真实产品链路中稳定完成，不以“模型输出了一个标签”为通过标准。

通过条件同时包含：

1. 业务意图理解正确；
2. 对话行为和当前上下文正确合并；
3. 路由到正确的数据或执行链路；
4. 最终回复满足用户本轮目标；
5. 不违反预算、地址、品类、忌口等硬约束。

## 2. 评测环境

- 页面：`http://127.0.0.1:5189/`
- 执行方式：在真实 Web 对话框逐条发送 Query
- 意图解析：LLM 与规则混合解析
- 模型：DeepSeek `deepseek-v4-flash`
- 数据：产品当前餐厅、菜品、用户画像和知识库
- 日期：2026-07-15
- 浏览器控制台错误：0

测试 Query 的结构化原始数据位于 `evals/business-intent-30-cases.json`。

## 3. 总体结论

| 结果 | 数量 | 占比 |

| 完整通过 | 10 | 33.3% |

## 5. 主要 Bug

### P0：候选引用没有版本和范围校验

“换一批”只返回1家以后，用户说“第二家”，系统不应继续执行。当前系统恢复旧候选，导致后续“第二个菜”又被误解为选择餐厅。

应增加：

- `candidateSetId`
- `candidateType`
- `candidateCount`
- 引用有效性检查
- 越界时针对性澄清

### P0：预算纠正没有覆盖旧状态

“不是每人150，是5个人合计150”被识别为纠正表达，但执行结果仍保留 `per_person`，继续计算总预算750元。

应采用显式状态操作：

```text
replace budget.scope: per_person -> total
replace budget.amount: 150 -> 150
```

更新后必须重新进行预算审计，不能复用旧规划结果。

### P0：地址意图只存在于语义层，没有执行闭环

地址 Query 能被部分语义规则看到，但实际仍走普通推荐；页面顶部地址保持“工作地”。

应建立：

```text
识别地址 -> 解析到已保存地址 -> 更新 deliveryContext
-> 清除旧候选与选择 -> 使用新地址重新召回
```

### P0：长期记忆缺少用户可见确认

三条记忆写入均进入 `memory_write_pending`，但最终显示“没有拿到可展示的回复”。状态存在不等于功能完成。

必须展示：

- 准备保存的记忆内容
- 记忆类型
- 使用范围
- 确认与取消操作
- 用户纠正后的撤销能力

### P1：硬约束被错误降级为相似条件

“烧烤”被近似成“辣”，返回川菜和川味面；“粥店”返回提供粥类商品但主营并非粥的餐厅。

品类、过敏、配送上限和明确预算应进入硬过滤。没有结果时应明确告知没有完全匹配项，不能用相似候选冒充满足。

### P1：餐厅详情与菜单查询边界错误

用户询问配送时间、距离、月销量时，系统直接返回菜单。需要根据用户目标区分：

- 查询餐厅事实：返回餐厅字段；
- 查询菜单：返回菜品列表；
- 选择餐厅：更新选择状态后返回菜单。

### P1：知识问答与即时推荐边界不稳定

“烧烤怎样点会少油”是方法咨询，却被“点”和“少油”触发为即时推荐。判断关键应是用户要“获得知识”还是“获得当前候选”，不能只靠关键词。

### P1：指定菜品搜索没有进入工具参数

“番茄鸡肉汤饭哪里有”已经给出明确菜品目标，但 `dishName` 没有驱动餐厅或菜单检索，最后返回无关餐厅。

### P2：运行体系仍未迁移到最新两级业务意图

开发者控制台仍展示“复杂点餐规划”“选店后的商品推荐”等旧标签。新两级业务意图目前只存在于产品讨论中，尚未成为运行 Schema、评测标签和路由输入。

## 6. 建议修复顺序

1. 将一期两级业务意图落到唯一运行 Schema；
2. 建立 `TaskState Reducer`，先修预算纠正和条件继承；
3. 建立候选集版本与指代门禁；
4. 打通地址更新执行链路；
5. 修复记忆确认和撤销闭环；
6. 将品类、过敏、配送和预算改为硬约束；
7. 修复餐厅详情、菜单、菜品搜索和知识问答边界；
8. 重跑本评测集，目标为30/30完整通过。
