# Step 3：基础工具调用层

这一阶段的目标是把“页面里的推荐逻辑”拆成一组 Agent 可调用的工具。后续接入真实大模型时，大模型不直接操作页面代码，而是选择调用这些工具。

## 为什么要做工具调用层

在真实 Agent 里，大模型负责判断“下一步该做什么”，代码负责执行“怎么做”。工具调用层就是两者之间的接口。

如果没有工具层，所有逻辑会混在 `app.js` 里：

- 难以接入大模型
- 难以替换真实接口
- 难以记录 Agent 到底做了哪些动作
- 难以做权限、安全确认和错误恢复

拆成工具后，每个能力都有明确名字、输入、输出和日志。

## 本步新增文件

### `tools.js`

核心意图：建立工具运行时，也就是一个简化版 tool dispatch map。

它负责：

- 定义工具函数
- 执行工具函数
- 记录工具调用轨迹
- 返回统一格式的结果

后续真实大模型接入时，可以把 `tools.js` 里的工具暴露给模型，由模型决定调用顺序。

## 本步新增或迁移的工具

### `get_user_memory`

用途：读取用户长期记忆。

产品意义：让推荐不只依赖当前输入，还能参考用户过往偏好。

未来接入：数据库或用户画像服务。

### `search_restaurants`

用途：根据用户需求和位置搜索候选餐厅。

产品意义：先从店铺层面过滤不合适的商家，例如配送时间太长的店。

未来接入：高德地图周边搜索、真实外卖商家接口。

### `get_menu`

用途：查询指定餐厅菜单。

产品意义：餐厅合适不代表菜品合适，必须继续看菜单。

未来接入：外卖平台菜单接口或自建菜单数据库。

### `rank_dishes`

用途：根据预算、配送时间、口味、忌口和长期记忆给菜品排序。

产品意义：这是点餐 Agent 的推荐决策层。

未来接入：规则排序 + 大模型解释 + 个性化推荐模型。

### `create_cart`

用途：把首选推荐变成下单前购物车草稿。

产品意义：Agent 可以帮用户准备订单，但不能直接下单，必须等待用户确认。

未来接入：订单草稿服务、下单确认流程。

## 本步新增关键函数

### `createToolRuntime()`

创建一套工具运行环境。它接收餐厅数据和用户画像，返回一组可调用工具。

产品经理理解：这是 Agent 的“工具箱”。

### `recordToolCall()`

每次工具调用后记录工具名、输入、状态和输出摘要。

产品经理理解：这是 Agent 的“行动日志”。没有行动日志，就很难知道 Agent 为什么给出某个推荐。

### `summarizeOutput()`

把工具返回的大结果压缩成一句可读摘要。

产品经理理解：这是可观测性设计。真实系统里，工具输出可能很长，前台应该优先展示摘要。

### `runAgentTools()`

在 `app.js` 中按顺序调用工具：

1. `get_user_memory`
2. `search_restaurants`
3. `get_menu`
4. `rank_dishes`
5. `create_cart`

产品经理理解：这是当前阶段的“手写 Workflow”。下一步 Step 4 会把它正式整理成固定工作流。

## 对应课程

- S02 Tool Use

S02 的核心思想是：循环不变，工具扩展。加能力时，不应该重写 Agent loop，而应该新增工具定义和工具处理函数。

当前 Step 3 的实现正是这个思想：

- `app.js` 保持页面和流程控制
- `tools.js` 负责工具能力
- 页面展示 `ToolCalls`，让工具调用过程可见
