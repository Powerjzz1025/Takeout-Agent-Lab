# 外卖智能点餐 Agent 任务叙事

## 1. Agent 名称

外卖点餐决策助理。

它不是普通餐厅列表，也不是菜单搜索框，而是帮助用户把模糊、临时、带约束的点餐需求转成“可比较、可选择、可继续决策”的 Agent。

## 2. 目标用户

核心用户是一线城市高频外卖用户，当前 Demo 用户画像是张三：

- 32 岁，北京互联网公司高级产品经理。
- 工作日午餐决策时间短。
- 偏好川菜、湘菜、中式烧烤、麻辣、香辣。
- 不吃香菜。
- 有控制体重诉求，每周至少 2 次轻食沙拉。
- 家庭点餐时需要考虑妻子忌食羊肉。

## 3. 典型业务场景

用户在午餐、晚餐或加班夜宵前打开点餐助手，说：

> 20 分钟内送到，清淡一点，预算 35 元左右。

或者说：

> 想吃川菜，但别太油，附近有什么合适的？

或者在上一轮推荐后说：

> 就第一家吧，看看有什么好吃的。

用户不是在要求系统展示所有商家，而是在委托 Agent 帮自己缩小选择范围。

## 4. Agent 角色

Agent 扮演“点餐决策助理”。

它的职责是：

- 理解用户真正想完成的点餐任务。
- 把自然语言需求转成结构化约束。
- 查询用户画像、长期记忆、餐厅数据、菜单数据和必要的饮食知识。
- 在配送、预算、口味、忌口、销量和场景之间做取舍。
- 先推荐 3 家最合适餐厅。
- 用户选店后，再推荐该店 5 个最匹配商品。
- 保存长期记忆前请求用户确认。

它不扮演医生、支付系统或真实外卖平台。本期 Demo 不提供购物车、下单和支付。

## 5. Agent 目标

Agent 的目标不是“执行流程”，而是：

帮助用户在尽量少的交互轮次内，先选出合适餐厅，再选出匹配诉求的商品。

一期 Demo 的可交付结果是：

- 一条由大模型中枢生成的自然语言回复。
- 3 家餐厅推荐。
- 用户选店后的 5 个商品推荐。
- 必要时的追问。
- 必要时的待确认长期记忆。

## 6. 输入信息

Agent 可以使用以下输入：

- 用户原始 query。
- 意图识别结果。
- 槽位：预算、配送时间、口味、忌口、人数、餐厅名、商品名、健康目标。
- 用户画像和长期记忆。
- Mock 餐厅数据。
- Mock 商品数据。
- RAG 知识库。
- 上一轮餐厅推荐结果。
- Skills 中定义的决策规则。
- 安全确认状态。

## 7. 行动链路

标准点餐链路：

1. 接收用户 query。
2. 使用 `takeout-intent-routing` Skill 判断意图和路由。
3. 抽取预算、配送时间、口味、忌口等槽位。
4. 如果关键信息不足，先追问。
5. 读取基础用户画像和相关长期记忆。
6. 查询 Mock 餐厅数据。
7. 必要时查询 RAG 知识库补充饮食原则。
8. 使用 `restaurant-ranking` Skill 对候选餐厅排序。
9. 返回 3 家餐厅推荐。
10. 用户选择某家餐厅。
11. 查询该店菜单。
12. 对该店商品排序，返回 5 个最匹配商品。
13. 大模型中枢整合工具、记忆、知识和 Workflow 状态，生成最终回复。

复杂点餐链路：

1. 识别多人、多约束或目标冲突。
2. 拆分每位成员的口味、忌口和预算。
3. 先生成更适合继续筛选的餐厅候选。
4. 用户选店后，再进入该店商品推荐。

记忆写入链路：

1. 用户表达稳定偏好或明确说“记住”。
2. Agent 生成待确认记忆。
3. 用户确认后，写入本轮长期记忆。

## 8. 决策规则

### 路由规则

- 点餐推荐：进入 `workflow`。
- 饮食知识：进入 `rag_lookup`，但知识库只作为大模型内部资料。
- 查附近餐厅或指定菜单：进入 `single_tool`。
- 保存偏好：进入 `memory_write`。
- 多人、多目标、多约束：进入 `planning`。
- 关键信息不足：进入 `clarify`。
- 模型身份、能力边界和泛问题：进入 `llm_direct`。

### 餐厅排序优先级

1. 忌口、过敏和明确不吃的内容。
2. 用户要求的配送时间。
3. 店铺距离。
4. 月售销量和评分。
5. 用户当前表达的口味目标。
6. 长期记忆中的稳定偏好。
7. 用餐场景，例如工作餐、晚餐、夜宵。
8. 核心餐品是否能承接用户需求。

### 商品排序优先级

1. 忌口、过敏和明确不吃的内容。
2. 商品价格是否贴近预算。
3. 商品规格是否适合当前人数和场景。
4. 月销量。
5. 口味和健康目标匹配度。
6. 长期记忆中的偏好和禁忌。
7. 商品描述是否符合用户表达的模糊诉求。

### 安全规则

- Agent 不创建购物车。
- Agent 不真实下单。
- Agent 不处理支付。
- Agent 可以生成待确认记忆，但不能静默保存敏感偏好。
- 涉及健康、过敏或疾病相关内容时，只做一般建议，并提醒用户自行确认。

## 9. 工具与能力

当前 Demo 具备：

- 意图识别。
- 槽位抽取。
- 缺槽追问。
- RAG 知识库查询。
- 餐厅搜索。
- 餐厅排序。
- 菜单查询。
- 商品排序。
- 复杂约束规划。
- 基础用户画像。
- 长期记忆雏形。
- 记忆保存确认。
- Skills 决策说明。
- DeepSeek 大模型中枢回复。

后续真实版本会接入：

- 高德地图地理编码、周边搜索和距离估算。
- 外卖平台或自建商家菜单接口。
- 服务端数据库。
- 评测与日志系统。
- 如未来要做下单，再单独设计购物车、订单、支付和安全确认。

## 10. 最终产出

Agent 每轮应该尽量产出以下一种或多种结果：

- 一句清晰回复。
- 3 家餐厅候选。
- 选店后的 5 个商品候选。
- 一个追问问题。
- 一个待确认长期记忆。
- 一个清晰的下一步动作建议。

## 11. 成功标准

一期 Demo 的成功标准：

- 能正确判断常见点餐意图。
- 能抽取预算、配送时间、口味、忌口和人数。
- 能在信息不足时追问，而不是盲目推荐。
- 能结合张三用户画像影响推荐。
- 用户表达点餐诉求后，先推荐 3 家餐厅。
- 用户选择餐厅后，再推荐 5 个商品。
- 能说明配送时间和距离是 Demo 模拟数据，不冒充真实地图 API。
- 能让大模型作为最终回复中枢，而不是直接裸露工具或知识库结果。
- 能在保存长期记忆前请求确认。

后续业务指标：

- 餐厅推荐采纳率。
- 商品推荐采纳率。
- 平均决策时间。
- 平均追问轮次。
- 约束满足率。
- 忌口和过敏误触发率。
- 用户确认率。

## 12. 失败与升级规则

Agent 应该承认无法完成，而不是强行推荐。

需要追问的情况：

- 同时缺少预算和配送时间，且需求非常模糊。
- 多人点餐缺少人数或主要约束。
- 指定菜单查询缺少餐厅名。

需要放宽条件的情况：

- 没有餐厅满足配送时间。
- 所有候选商品都超预算。
- 所有候选商品都碰到忌口或过敏。

需要用户确认的情况：

- 保存敏感偏好。
- 修改长期记忆。
- 健康、过敏或疾病相关高风险建议。

## 13. 一期 Demo 边界

当前版本只做 Web Demo。

- 不真实下单。
- 不真实支付。
- 不调用真实外卖平台。
- 不调用高德地图实时接口。
- 不把浏览器内存中的新增记忆持久化到数据库。
- 不把饮食建议包装成医疗建议。

这个边界让一期产品先验证 Agent 任务理解、工具链路、推荐逻辑、长期记忆和大模型中枢，再进入真实 API 接入。
