# Step 7：用户画像与长期记忆

这一阶段的目标是让 Agent 拥有一个可复用的用户画像，并能基于当前需求检索相关长期记忆。

本项目里的用户画像是虚拟 Demo 用户，不代表真实个人信息。

## 为什么要做用户画像

外卖点餐助手如果每次都从零开始问用户，会很像普通搜索框。用户画像能让它更像私人助手：

- 知道常用预算
- 知道口味偏好
- 知道不吃什么
- 知道常用配送地点
- 知道工作日午餐更重视速度和稳定

这些信息可以帮助 Agent 更快、更准地推荐。

## 本步新增和更新的文件

### `data/user-profile.json`

核心意图：保存完整 Demo 用户画像和长期记忆。

本步扩展为：

- `profile`：身份、年龄段、收入段、生活阶段、工作节奏
- `locations`：工作地、住处、配送偏好
- `preferenceSummary`：口味、品类、预算、配送和下单习惯摘要
- `memories`：长期记忆
- `pendingMemories`：待确认记忆

产品经理理解：这是用户画像数据源。它让 Agent 从“问答工具”变成“懂用户上下文的助手”。

### `memory.js`

核心意图：建立长期记忆运行时。

用于做什么：

- 生成用户画像快照
- 根据当前需求检索相关记忆
- 生成记忆摘要
- 创建待确认记忆

Agent 架构位置：Long-term Memory Layer。

未来接入：数据库、用户授权、记忆写入确认、记忆过期和置信度更新。

## 本步新增关键函数

### `createMemoryRuntime()`

核心意图：创建记忆运行时。

用于做什么：接收用户画像，返回读取画像和检索记忆的方法。

产品经理理解：这是长期记忆服务的入口。

### `getProfileSnapshot()`

核心意图：返回用户画像快照。

用于做什么：给页面和 Agent 提供当前用户基本画像、常用地点和偏好摘要。

产品经理理解：这是 Agent 对“这个用户是谁”的当前理解。

### `getRelevantMemories()`

核心意图：按当前需求检索相关长期记忆。

用于做什么：不是每次把所有记忆都塞给 Agent，而是按口味、预算、意图、场景取最相关的几条。

产品经理理解：长期记忆也需要检索。记忆太多时，全部塞进上下文会让模型混乱。

### `scoreMemory()`

核心意图：给记忆相关性打分。

用于做什么：预算缺失时预算记忆更重要；工作餐场景下配送和午餐习惯更重要；忌口记忆始终重要。

产品经理理解：这是记忆版的排序逻辑。

### `buildMemorySummary()`

核心意图：把命中的记忆压缩成摘要。

用于做什么：给工具调用日志和后续大模型上下文使用。

产品经理理解：大模型不一定需要完整记忆列表，有时一段摘要更清晰。

### `createPendingMemory()`

核心意图：创建待确认记忆。

用于做什么：当用户说“记住我不吃香菜”时，先生成待确认记忆，不直接静默写入长期记忆。

产品经理理解：记忆写入需要谨慎。尤其是地址、健康、过敏、收入等敏感或半敏感信息，应先确认。

## 记忆分层

### 基础画像

相对稳定的信息，例如身份、年龄段、生活阶段。

### 偏好摘要

长期稳定但可变化的信息，例如口味、预算、常用品类。

### 长期记忆

对未来任务有用的事实或偏好，例如“不吃香菜”“午餐预算 30-40”。

### 待确认记忆

用户刚提出、还没有确认保存的信息。

- `UserProfile` 显示完整用户画像
- `UserMemory` 命中预算、清淡偏好、忌口和午餐习惯
- `ToolCalls` 中的 `get_user_memory` 会返回记忆摘要

```text

```

预期：

- `IntentResult.route` 是 `memory_write`
- `ToolCalls` 包含 `save_user_memory`
- `UserMemory` 中出现待确认记忆
