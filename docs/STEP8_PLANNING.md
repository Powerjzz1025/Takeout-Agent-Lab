# Step 8：复杂任务规划

这一阶段的目标是处理多人、多约束、多目标的点餐需求。

示例：

```text
两个人，一个不吃辣，一个想高蛋白，预算 80
```

这类需求不能只用普通推荐 Workflow，因为它包含多个子目标。

## 为什么需要 Planning

普通 Workflow 适合稳定链路：

```text
解析需求 -> 查记忆 -> 查知识 -> 搜餐厅 -> 排序 -> 购物车
```

复杂任务 Planning 需要先拆任务：

```text
识别人群 -> 拆分每个人约束 -> 分预算 -> 分别推荐 -> 合并购物车
```

产品经理理解：Workflow 是固定业务流程，Planning 是把复杂目标拆成可执行步骤。

## 本步新增文件

### `planning.js`

核心意图：建立复杂点餐规划层。

用于做什么：

- 识别参与人数
- 为每个人拆分口味和忌口
- 分配人均预算
- 分别检索知识和排序菜品
- 合并多人购物车

Agent 架构位置：Planning / Task Decomposition。

未来接入：ReAct 式动态规划、大模型 planner、多 Agent 分工。

## 本步新增工具

### `create_group_cart`

核心意图：为多人点餐创建购物车草稿。

用于做什么：把多个成员的推荐菜品合并成一个多人订单草稿。

产品经理理解：多人点餐的结果不是一个菜，而是一组满足不同约束的菜品组合。

## 本步关键函数

### `createPlanningRuntime()`

核心意图：创建复杂任务规划运行时。

用于做什么：提供 `planComplexOrder()` 方法。

### `planComplexOrder()`

核心意图：执行完整复杂点餐规划。

流程：

1. 拆分成员需求
2. 生成规划步骤
3. 搜索共享候选餐厅
4. 读取长期记忆
5. 为每个成员检索知识库
6. 为每个成员排序菜品
7. 合并购物车
8. 输出规划摘要

### `buildParticipantNeeds()`

核心意图：把一句复杂 query 拆成多个成员的需求。

当前支持：

- `一个不吃辣`
- `一个想高蛋白`
- `两个人`
- 总预算拆成人均预算

产品经理理解：这一步就是把“一个混合需求”拆成“多个可执行需求”。

### `buildPlanningSteps()`

核心意图：生成可视化规划步骤。

用于做什么：让页面展示系统如何拆任务。

### `buildPlanningSummary()`

核心意图：把规划结果变成用户能读懂的总结。

用于做什么：说明每位成员推荐了什么、合计多少钱、是否等待确认。

## 页面新增状态

### `PlanningState`

用于展示：

- 规划状态
- 人数
- 总预算
- 人均预算
- 规划步骤
- 每个成员的推荐
- 多人购物车草稿

## 当前局限

当前 Planning 仍然是规则版：

- 只能识别少量中文表达
- 只支持简单两人拆分
- 多人组合仍然比较粗糙
- 没有真正的大模型动态规划

后续升级方向：

- 用大模型输出结构化计划
- 支持更多成员约束
- 支持同店组合优化
- 支持预算、配送费、满减综合优化
- 支持计划失败后的重试和放宽约束

预期：

- `IntentResult.route` 是 `planning`
- `PlanningState.peopleCount` 是 `2`
- `PlanningState.steps` 展示拆分步骤
- `ToolCalls` 包含 `create_group_cart`
- `Cart.mode` 是 `group`
