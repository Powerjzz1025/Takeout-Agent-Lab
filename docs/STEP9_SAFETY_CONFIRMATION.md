# Step 9：安全确认

这一阶段的目标是把关键动作从“Agent 自动执行”改成“Agent 准备，用户确认后执行”。

## 为什么需要安全确认

外卖 Agent 会涉及一些有风险的动作：

- 保存长期记忆
- 保存地址、偏好、过敏等敏感或半敏感信息
- 确认购物车
- 提交订单
- 支付

这些动作不能让 Agent 自动越权执行。Agent 可以建议和准备，但用户必须最终确认。

## 本步新增文件

### `safety.js`

核心意图：建立安全确认运行时。

用于做什么：

- 根据 Workflow 结果生成待确认动作
- 判断动作风险级别
- 处理用户确认或取消
- 更新购物车状态或长期记忆状态
- 记录用户决策

Agent 架构位置：Safety / Human Approval Layer。

未来接入：真实下单、支付、权限、风控、审计日志。

## 本步新增页面模块

### `待确认动作`

页面会显示需要用户批准的动作。

当前包括：

- 确认购物车草稿
- 确认多人购物车草稿
- 确认保存长期记忆

每个动作都有：

- 标题
- 描述
- 风险级别
- 确认按钮
- 取消按钮

### `SafetyState`

用于展示：

- `pendingActions`：等待确认的动作
- `decisions`：已经批准或拒绝的动作记录

## 本步关键函数

### `createSafetyRuntime()`

核心意图：创建安全确认运行时。

用于做什么：提供待确认动作生成和用户决策处理能力。

### `buildPendingActions()`

核心意图：根据 Agent 结果生成待确认动作。

当前规则：

- 如果购物车状态是 `needs_confirmation`，生成订单确认动作
- 如果记忆状态是 `pending_confirmation`，生成记忆保存确认动作

产品经理理解：Agent 的输出结果不等于最终执行。中间要经过安全层。

### `resolveAction()`

核心意图：处理用户确认或取消。

用于做什么：

- 用户确认购物车 -> 购物车状态变成 `confirmed`
- 用户确认记忆 -> 待确认记忆变成长期记忆
- 用户取消 -> 记录拒绝，不执行动作

### `buildDecisionRecord()`

核心意图：记录用户决策。

用于做什么：保留动作 ID、类型、风险级别、批准/拒绝结果。

产品经理理解：真实系统里这会进入审计日志。

## 当前实现边界

当前仍是前端 Demo：

- 购物车确认只是在页面状态中变为 `confirmed`
- 记忆确认只在当前浏览器会话中加入内存里的用户画像
- 不会真实下单
- 不会真实支付
- 不会写回本地 JSON 文件

后续接服务端后，可以把确认动作真正写入数据库。

## 你可以如何验证

输入：

```text
20 分钟内送到，清淡一点，预算 35 元左右
```

预期：

- 生成购物车草稿
- 页面出现“确认购物车草稿”
- 点击确认后，`Cart.status` 变成 `confirmed`

输入：

```text
记住我不吃香菜
```

预期：

- 生成待确认记忆
- 页面出现“确认保存长期记忆”
- 点击确认后，记忆进入长期记忆
