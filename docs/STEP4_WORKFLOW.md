# Step 4：固定点餐 Workflow

这一阶段的目标是把 Step 3 的工具串成一个稳定业务流程。工具解决“能做什么”，Workflow 解决“什么时候做、按什么顺序做、遇到信息不足怎么办”。

## 为什么需要 Workflow

如果只有工具，Agent 仍然可能乱用工具。例如用户只说“随便推荐点”，系统直接搜索餐厅，就很可能给出不稳定结果。

Workflow 的作用是先定义可靠路径：

1. 解析用户需求
2. 判断是否需要追问
3. 读取长期记忆
4. 搜索候选餐厅
5. 查询菜单并排序菜品
6. 生成购物车草稿
7. 等待用户确认

这条路径让 Demo 更像真实产品，而不是一个只会聊天的页面。

## 本步关键函数

### `parseNeed()`

核心意图：把用户自然语言转成结构化需求。

本步变化：从 `app.js` 移到 `workflow.js`。

为什么移动：解析需求属于业务流程的一部分，不属于页面展示逻辑。

产品经理理解：用户输入是非结构化的，产品执行必须结构化。这个函数就是从“人话”到“系统字段”的入口。

### `createOrderingWorkflow()`

核心意图：创建点餐 Workflow。

用于做什么：把工具和用户画像组合起来，返回一个可以执行的 `run()` 方法。

产品经理理解：这是点餐 Agent 的业务主线。它决定从用户输入到购物车草稿之间发生什么。

### `shouldAskClarifyingQuestion()`

核心意图：判断是否需要追问。

用于做什么：如果用户输入太模糊，例如“随便推荐点”，并且缺少预算和配送时间，就先追问。

产品经理理解：优秀 Agent 不应该一味回答。信息不足时，先问一个关键问题，往往比硬推荐更可靠。

### `buildWorkflowState()`

核心意图：生成可视化流程状态。

用于做什么：记录每个流程节点是完成、等待、阻塞还是待执行。

产品经理理解：这是 Agent 的流程看板。它让用户和产品团队都能看到 Agent 当前走到哪一步。

### `run()`

核心意图：执行完整点餐流程。

用于做什么：

1. 重置工具调用日志
2. 解析用户需求
3. 判断是否追问
4. 调用记忆、餐厅、菜单、排序、购物车工具
5. 返回推荐、购物车、工具调用轨迹和流程状态

产品经理理解：这是当前阶段的“固定 Workflow”。它比完全自主 Agent 更可控，适合一期 Demo。

### `buildReply()`

核心意图：把流程执行结果转成用户能理解的回复。

用于做什么：解释推荐结果、假设条件和购物车状态。

产品经理理解：用户不关心工具怎么跑，用户关心“为什么推荐这个、下一步能做什么”。

## Workflow 和 Agent 的区别

Workflow 是固定路径：

- 适合高频、标准、风险可控的产品链路
- 优点是稳定、可测试、可解释
- 缺点是灵活性有限

Agent 是动态决策：

- 适合开放问题、多步探索、路径不确定的任务
- 优点是灵活
- 缺点是更难控制和评测

一期外卖 Demo 应该优先使用 Workflow。等基础链路稳定后，再在复杂请求中加入 ReAct 式动态规划。

## 你可以如何验证

输入明确需求：

```text
20 分钟内送到，清淡一点，预算 35 元左右
```

预期结果：

- 会生成推荐
- 会生成购物车草稿
- `WorkflowState.status` 是 `ready_for_confirmation`

输入模糊需求：

```text
随便推荐点
```

预期结果：

- 不会直接生成推荐
- 会追问预算或配送时间
- `WorkflowState.status` 是 `needs_clarification`
