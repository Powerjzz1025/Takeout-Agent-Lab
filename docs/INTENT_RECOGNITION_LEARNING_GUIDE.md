# AI Agent 意图识别与意图体系搭建学习指南

> 适用对象：第一次设计 Agent 产品的 AI 产品经理  
> 学习目标：能独立完成意图盘点、分层、边界定义、槽位设计、多轮状态处理、识别策略和评测方案  
> 内容来源：课堂笔记、课程截图、当前外卖点餐 Agent 的真实 bad case，以及主流对话系统设计方法

## 1. 先建立正确的基本概念

在 Agent 中，意图设计不是给一句话贴一个标签，而是建立从用户表达到账户、数据、工具和 Workflow 的稳定映射。

```text
Query  = 用户表面说了什么
Intent = 用户希望达成什么业务结果
Slot   = 完成目标所需的结构化信息
State  = 多轮对话中当前仍然有效的任务事实
Action = 系统下一步执行什么
```

例如：

```text
Query: “20 分钟内送到，清淡一点，预算 35 元。”
Intent: 找到适合当前一餐的餐厅候选
Slots: 配送上限=20、口味=清淡、预算=35
State: 当前地址、历史已确认约束、已展示候选
Action: 搜索商家 -> 硬约束过滤 -> 排序 -> 展示三家
```

Intent 和 Action 不能直接画等号。相同 Intent 可能因为槽位、状态、权限和风险不同，进入不同 Action；同一个 Tool 也可能服务多个 Intent。

## 2. PM 真正需要设计的不是一个列表

每个可上线的意图至少需要一张“意图卡片”：

| 字段 | PM 要回答的问题 |
| --- | --- |
| Intent ID | 系统内部如何稳定命名 |
| 中文名称 | 产品、算法、研发和运营能否共同理解 |
| 用户目标 | 用户最终想得到什么结果 |
| 适用旅程 | 发生在下单前、决策中还是售后 |
| 显式表达 | 用户直接说出目标时怎么讲 |
| 隐式表达 | 用户不说标准关键词时怎么讲 |
| 必填槽位 | 缺少哪些信息绝对不能执行 |
| 可继承槽位 | 哪些信息可从 State、画像或系统数据获得 |
| 边界 | 什么属于它，什么不属于它 |
| 易混淆意图 | 最容易和谁混淆，区分信号是什么 |
| 承接动作 | 回复、检索、Tool、Workflow、Planning 或人工 |
| 风险等级 | 误判是否造成资金、安全或数据影响 |
| 澄清策略 | 不确定时只问哪个最有信息量的问题 |
| 失败策略 | 无结果、Tool 失败、能力不支持时怎么办 |
| 评测样例 | 正例、近邻反例、隐式表达、多轮和多意图样例 |

只写 Intent 名称和几个标准 Query，不能称为完整意图设计。

## 3. 为什么真实意图识别很难

课程总结了五类核心难点，外卖 Agent 中全部存在。

### 3.1 意图数量多

当分类器一次面对几十或上百个标签时，标签说明、few-shot 和上下文会互相干扰。候选越多，相近标签越容易混淆。

典型解决办法：

- 层次分类：先业务域，再任务族，最后原子目标；
- 检索增强：先召回最相关的 Top-K 候选 Intent，再让模型精分类；
- 动态 few-shot：只给模型当前候选附近的正例和近邻反例。

### 3.2 意图边界重叠

例如“第一家和第二家哪个好”同时包含比较和推荐含义。解决问题不能只靠改标签名称，而要先判断两类是否会产生不同业务结果。

处理原则：

1. 如果两种解释会触发不同工具、数据或风险动作，必须拆开并定义边界；
2. 如果用户无论如何都能接受同一个结果，优先合并；
3. 如果差异只是对象或参数，保留一个动作，用 Target/Slot 区分；
4. 每对近邻 Intent 都必须准备反例和边界测试。

### 3.3 用户表达口语化

用户不会按照意图名称说话：

```text
“还是刚才那种，但是快一点。”
“这几家没感觉。”
“第二个吧。”
“今天胃有点顶，不想吃太刺激的。”
```

这类输入需要上下文、指代消解、行业词汇和真实语料，单靠关键词规则很难稳定完成。

### 3.4 一句话包含多个目标

```text
“以后都不要香菜，今天帮我找一家清淡的。”
```

它包含当餐推荐和长期记忆更新，但不应该简单拆成两个平行 Agent。更合理的结构是：

```text
Primary Goal: 当前餐厅推荐
Post Action: 提议保存长期忌口，等待用户确认
```

多意图识别要先判断：用户是否真的需要多个独立结果，还是只是在描述完成一个结果所需的过程。

### 3.5 多轮对话依赖状态

```text
Turn 1: “公司五个人，预算 150。”
Turn 2: “合计。”
Turn 3: “错了，每个人。”
```

第二、三轮单独看都不是完整 Intent。系统必须继承当前 Goal，并把本轮解析成对槽位的 `set / replace / remove` 操作。

很多所谓“模型理解不好”，本质上是 State、候选集版本和上下文管理没有设计好。

## 4. 从业务流程搭建意图体系

### 4.1 先确定产品边界

不要直接照搬行业全旅程。完整外卖平台可能包含：

```text
下单前：找店、找菜、比较、优惠、评价
下单中：确认地址、规格、支付
下单后：配送查询、催单、联系骑手、取消、退款、投诉
复购：再来一单、收藏、评价、发票
```

当前项目是“下单前的点餐决策助手”，没有真实订单、支付、骑手和售后系统。因此取消订单、催单、退款、投诉不能伪装成已支持 Intent；它们应被识别为目标清楚但 `unsupported`。

### 4.2 从用户结果反推 Intent

判断是否需要单独 Intent 的首要问题是：**用户期望的最终输出是否不同？**

- 输出餐厅列表：餐厅发现；
- 输出两家店的差异：餐厅比较；
- 输出某家店的菜单：菜品浏览；
- 输出完整多人搭配：用餐方案；
- 输出饮食知识：知识问答。

不要因为实现方式不同就创建 Intent。`single_tool / workflow / planning / rag` 都是执行策略，不是用户目标。

### 4.3 再判断是否需要分层

满足以下任一条件时应使用层级意图：

- 叶子 Intent 超过约 15-20 个；
- 存在多个具有共同上位语义的近邻 Intent；
- 不同任务族拥有不同上下文、槽位或候选数据；
- 希望先缩小候选集，再做精分类；
- 需要分层统计和分层评测。

层级不是越深越好。每增加一级都会增加延迟、错误传播和维护成本。

## 5. 本项目为什么采用三级

当前 V2.1 已有 Domain 和 Goal 两级，但 `compare / inspect / explain / sort` 的对象被压进扁平 Goal：

```text
order.explain_recommendation
order.sort_candidates
```

系统无法仅从标签判断是在解释餐厅还是解释菜品，也无法判断排序的是餐厅集还是菜品集。

因此采用三级组合：

```text
L1 Domain：属于什么业务域
L2 Task Family：操作什么业务对象或任务族
L3 Atomic Goal：希望对该对象完成什么结果
```

例如：

```text
ordering.restaurant.compare
ordering.restaurant.explain
ordering.dish.compare
ordering.dish.explain
```

三级已经足够。第四级不应该继续放 `refine / correct / confirm / route`，这些是正交字段。

## 6. MECE 不只是“看起来整齐”

### 6.1 相互独立 Mutually Exclusive

同一层必须使用同一个分类标准：

- L1 全部按业务域分；
- L2 全部按业务对象/任务族分；
- L3 全部按用户希望获得的原子结果分。

不能一半按对象分，一半按工具分；不能把 `unknown`、`complex`、`single_tool` 和业务目标并列。

### 6.2 完全穷尽 Collectively Exhaustive

穷尽并不等于给全世界每个需求建 Intent，而是：

- 产品范围内的高频目标都有明确叶子节点；
- 范围外但能理解的目标进入 `other.out_of_scope.respond`；
- 没理解的输入进入 Understanding Status，而不是 `unknown Intent`；
- 多意图通过 Primary + Supporting/Post Actions 表达。

## 7. 意图、槽位和状态的边界

### 7.1 Slot 不是 Intent

“清淡、35 元、20 分钟、五个人”都是约束信息，不是独立 Intent。

槽位应至少保存：

```text
value
source
confidence
strictness
scope
appliesTo
updatedAtTurn
```

### 7.2 当前任务信息进入 State

```text
当前地址
当前预算和范围
配送时限
口味与忌口
参与者及成员级约束
已展示餐厅/菜品候选集版本
当前选择
待确认动作
```

这些信息每轮更新，不应该每轮都重新询问。

### 7.3 长期信息才进入 Memory

适合长期保存：

- 用户明确表达的稳定偏好；
- 稳定事实和默认地址；
- 对未来有长期价值的安全约束；
- 已完成任务的重要结果；
- 用户明确要求保存的选择。

不适合直接写长期记忆：

- “今天想吃辣”；
- 某次临时地址；
- 未确认的模型推测；
- 一次候选列表中的浏览行为。

推荐结构：

```text
原始对话：保存可追溯原文
Task State：维护当前任务参数
Event Memory：保存完成任务的重要事件
Long-term Memory：保存稳定事实和偏好
```

记忆提取和 Intent 识别有关联，但不是同一任务。Intent 决定用户在做什么，Memory Policy 决定哪些信息未来值得保留。

## 8. 多意图的正确处理

### 8.1 输出结构

```json
{
  "primaryIntent": "ordering.restaurant.discover",
  "contextUpdates": [],
  "supportingTasks": [],
  "postActions": [
    {
      "type": "memory.propose_upsert",
      "requiresConfirmation": true
    }
  ]
}
```

### 8.2 执行关系

- 有依赖：先执行前置任务，再执行主目标；
- 无依赖且都必要：可以并行；
- 用户只关心一个最终结果：其他内容作为内部步骤；
- 高风险副作用：等待确认；
- 多目标复杂且有依赖：进入显式 Plan。

不要为了展示 Agent 很复杂，就把一句话拆成很多用户不关心的子任务。

## 9. 推荐的识别架构

```text
Query + Session State + 候选集上下文
  -> 规则提取确定性实体和高风险信号
  -> 意图候选检索（Top-K）
  -> 动态装配相关 Intent 定义、边界、正反例
  -> LLM 分层识别 Domain / Family / Goal
  -> Schema 校验与确定性 Policy 复核
  -> 槽位 State Reducer
  -> Clarify / Workflow / Tool / RAG / Planning
```

### 9.1 候选检索

Intent 很多时，可以使用关键词、BM25 或向量检索先召回候选。检索阶段目标是高 Recall，不负责最终决策。

当前产品的叶子数量不算巨大，第一期可以先做 Domain/Family 规则缩小范围；随着 Intent 增长，再增加向量检索。

### 9.2 动态 few-shot

Prompt 中固定保留：

- Schema；
- 产品边界；
- 未理解/不支持处理原则；
- 高风险规则。

动态加入：

- 当前候选 Intent 的定义；
- 与 Query 最相关的正例；
- 最容易混淆的近邻反例；
- 当前任务状态和候选集摘要。

不要把所有 Intent 的所有例子永久塞进一个 Prompt。

### 9.3 规则、LLM 和检索的职责

```text
规则：格式、数值、候选序号、高风险词和 Schema 校验
检索：缩小 Intent 与 few-shot 候选范围
LLM：理解口语、隐含表达、多轮关系和语义冲突
Policy：决定支持边界、状态变更、澄清和执行策略
```

## 10. 置信度、澄清和兜底

### 10.1 不要迷信模型自报置信度

不同识别器给出的 0.8 不一定含义相同。多识别器加权前必须在同一标注集上做校准，否则简单相加没有统计意义。

更可靠的决策信号：

- Top-1 是否超过该任务族的校准阈值；
- Top-1 与 Top-2 的 Margin；
- 是否存在未绑定候选引用；
- 是否缺少阻塞槽位；
- 是否命中高风险动作；
- 下游工具是否能验证关键事实。

### 10.2 澄清策略

只问一个信息增益最高的问题：

```text
错误：“请告诉我预算、人数、时间、口味和地址。”
正确：“你说的 150 元是五个人合计，还是每人 150 元？”
```

### 10.3 不识别后回滚

课程提出可以先按照最可能 Intent 执行，发现错误再回滚。这个方法只适用于：

- 只读、可逆、低成本操作；
- 不涉及资金、安全、隐私和长期写入；
- 用户能快速观察并纠正结果。

支付、删除记忆、真实下单、取消订单、修改订单等高风险操作必须先澄清或确认。

### 10.4 多识别器仲裁

可以组合规则、向量匹配和 LLM，但必须先定义：

- 每个识别器擅长什么；
- 分数如何校准；
- 哪些高风险规则拥有否决权；
- 分歧时是澄清、复核还是选择最高分；
- 仲裁结果如何记录和评测。

## 11. 意图边界设计方法

每对易混淆 Intent 都写一张边界表：

| A | B | 核心区分 |
| --- | --- | --- |
| 餐厅发现 | 菜品浏览 | 输出是餐厅列表，还是指定店菜单 |
| 餐厅比较 | 餐厅解释 | 比较多个候选，还是解释一个推荐依据 |
| 餐厅发现 | 完整用餐方案 | 输出候选店，还是成员级菜品组合 |
| 知识问答 | 当餐推荐 | 只要通用知识，还是要立即获得候选 |
| 记忆更新 | 当餐约束 | 长期保存是主要目标，还是只影响当前一餐 |
| 未理解 | 不支持 | 没听懂目标，还是听懂但产品做不了 |

## 12. 对课堂笔记示例的必要校正

以下表达不能脱离状态直接定 Intent：

| 表达 | 不能直接判断的原因 |
| --- | --- |
| “这个还能不要了吗？” | 可能是删除未下单商品、取消订单或退款 |
| “我点错了。” | 下单前可能是修改选择，下单后才可能取消 |
| “我不想吃这家了。” | 可能只是换候选，也可能涉及已下订单售后 |
| “怎么这么久？” | 可能查询进度，也可能表达抱怨；是否要求采取动作决定是否催单 |
| “我想吃麻辣烫。” | 麻辣烫是 `dishTarget/category`，产品当前输出仍可能是餐厅候选 |

Intent 的判定必须结合任务阶段、订单状态、当前候选和用户期望的下一步结果。

## 13. 评测体系

### 13.1 离线数据分层

1. Domain 分类集；
2. Family 分类集；
3. Leaf Goal 分类集；
4. Slot 与 Slot Operation 集；
5. 多轮上下文和指代集；
6. Multi-intent 与任务依赖集；
7. 未理解、不支持和社交边界集；
8. 高风险与安全集；
9. 真实口语、错别字和行业表达集；
10. 历史线上 bad case 回归集。

### 13.2 核心指标

- 各层 Accuracy 与 Macro-F1；
- 每个叶子 Intent 的 Precision/Recall；
- 近邻 Intent 混淆矩阵；
- Top-K 候选召回率；
- 槽位抽取 F1；
- 槽位 `set/replace/remove` 准确率；
- 上下文继承与候选引用准确率；
- 澄清触发准确率和平均澄清轮次；
- Action/Task Success Rate；
- 高风险误执行率；
- 未支持能力的诚实边界率。

总体 Accuracy 高不能证明系统可靠。低频类别全部错掉时，总体准确率仍可能很好，因此必须看 Macro-F1、分类别指标和高风险门禁。

## 14. 常见反模式

- 把 Tool、Route、Workflow 名称当 Intent；
- 把“复杂任务”当意图，而不看用户最终产出；
- 把 `unknown` 和社交合在一起；
- 把低置信度当成业务类别；
- 用槽位差异制造大量 Intent；
- 一句话只允许一个标签，丢掉记忆或地址副作用；
- 每轮重新询问已经存在于 State 的参数；
- 只写标准表达，不收集隐含表达和近邻反例；
- 只看几十条人工试用，不建立固定评测集；
- 识别后直接执行高风险动作，不做 Policy 和确认。

## 15. PM 的标准工作流

```text
1. 明确产品能力边界
2. 按用户旅程收集真实任务
3. 用“期望输出”归并用户目标
4. 建立 Domain -> Family -> Atomic Goal
5. 为每个叶子意图填写意图卡片
6. 定义共享槽位和成员级槽位
7. 编写近邻边界矩阵
8. 设计 State、候选引用和记忆策略
9. 定义 Primary/Secondary 和依赖关系
10. 设计澄清、兜底和高风险确认
11. 建立分层标注集和发布门禁
12. 影子运行，收集 bad case，持续回归
```
