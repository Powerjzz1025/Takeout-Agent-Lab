## 补充修复：短期会话记忆没有继承上一轮点餐条件

### 问题

用户先说：

> 20 分钟内送到，清淡一点，预算 35 元左右。

系统推荐餐厅后，用户又说：

> 还是重口味一些

或：

> 和刚才一样

系统却重新追问预算和配送时间。

### 原因

当前产品里有三种“记忆”：

1. 长期记忆：`data/user-profile.json` 和 `memory.js`，记录用户画像、口味偏好、忌口、预算习惯。
2. 短期选择记忆：`selectedRestaurant`、`selectedDish`，用于用户选店、选商品。
3. 对话上下文摘要：`context-compact.js`，用于展示和后续扩展。

但之前缺少第四层：普通追问/改口场景的“上一轮点餐条件继承”。

也就是说，系统知道用户刚才说过预算和配送时间，但没有把这些条件重新送回 `intent.js + workflow.js` 的规则理解层。规则层只看到孤立的“还是重口味一些”，所以会认为预算、配送时间缺失。

### 修复

- `app.js` 新增 `resolveContextualPrompt()`。
- 当用户说“还是、继续、和刚才一样、换成、改成、重口、清淡”等追问或改口表达时，会继承上一轮 `userNeed` 中未被否定的条件。
- 会继承预算、配送时间、口味、忌口。
- 如果用户明确给了新的预算、配送时间、口味或忌口，就以当前输入为准。
- `buildLLMReviewPayload()` 增加 `contextualText` 和 `shortTermContext`，让大模型复核时也能看到规则层实际理解的上下文。

### 当前理解链路

这个 Demo 当前不是纯 LLM Agent，而是混合架构：

- 意图识别、槽位抽取、路由、工具调用和推荐排序主要靠规则代码。
- DeepSeek 大模型目前作为“中枢复核和最终表达层”。
- RAG、Memory、Workflow、Tools 给大模型提供结构化状态。

所以，如果规则层没有拿到上下文，大模型最终也容易跟着问错问题。真正可靠的 Agent 需要让“短期上下文 -> 结构化状态 -> 工具调用 -> 大模型表达”全部对齐。

### 产品经理理解

长期记忆解决的是“这个用户平时喜欢什么”。短期会话记忆解决的是“刚才这一轮对话已经确认了什么”。两者不能互相替代。

这次问题说明：外卖 Agent 里，用户的预算、配送时间、当前口味改动、当前选店选品，都应该作为短期任务状态保留，而不是每一轮重新从单句 query 开始理解。

### 背后的 Agent 原理

一个 Agent 不是“一个大模型回复框”。更接近行业实践的结构是：先有 Agent Loop 负责循环推进任务，再由 Intent、Workflow、Tools、Memory、RAG、Skills 等模块提供判断和行动能力。Permission 决定边界，Hooks 记录过程，Todo 展示任务计划，Subagent 做局部审查，Context Compact 解决长会话上下文膨胀，System Prompt 约束大模型如何作为中枢整合结果。

### 产品经理理解

你可以把 S01-S10 理解成一个可工作的 Agent 最小骨架：

1. S01 让 Agent 有完整任务循环。
2. S02 让 Agent 能调用外部能力。
3. S03 让 Agent 知道什么能做、什么不能做。
4. S04 让关键过程可追踪。
5. S05 让复杂任务有可见步骤。
6. S06 让局部审查角色帮主 Agent 发现问题。
7. S07 让 Agent 在特定任务中加载稳定方法论。
8. S08 让长对话不会无限膨胀。
9. S09 让用户偏好可以长期沉淀。
10. S10 让大模型知道自己的角色、边界和输出标准。

本项目当前已经具备这 10 个环节的一期 Demo 实现。它们都不是最终生产形态，但已经足够支撑你理解一个 Agent 产品从“用户输入”到“结构化判断、调用工具、记忆约束、大模型生成”的完整路径。

## 最新更新：Step 14 切换到 OpenAI GPT 大模型

### 补充修复：清淡需求推荐不匹配

问题：

- 用户明确说“清淡一点”，系统仍然推荐川菜、麻辣、香辣类餐厅。
- 用户选择“轻卡计划沙拉碗”后，商品推荐却来自“蜀巷小碗菜”。

原因：

- `rank_restaurants()` 之前把清淡当作普通加分项，距离、销量和张三川味偏好会把重口餐厅抬上来。
- `findRestaurantByName()` 有一个匹配条件写错，导致输入任意餐厅名时可能命中第一家餐厅。
- DeepSeek 最终表达时会自己解释“虽然不匹配但距离近”，这会削弱工具层排序结果。

修复：

- `tools.js` 中清淡、少油、低脂、轻食、粥、蒸菜变成强约束。
- 明显重口的麻辣、香辣、川味、湘菜、烧烤、小龙虾餐厅在清淡需求下大幅降权。
- 新增 `清和蒸菜馆` Mock 餐厅，补足清淡热食候选。
- 修复 `findRestaurantByName()`，餐厅名先做精确和规范化匹配，不再误命中第一家。
- `app.js` 在用户说“第一家/第二家/第三家”时继承上一轮预算、配送时间、口味和忌口。
- `server.js` 增加结构化推荐回复护栏：餐厅和商品推荐只从工具返回的数据生成，不让大模型编造候选。

产品经理理解：

推荐系统里不是所有条件权重都一样。用户明确说出的口味约束，比如“清淡”“不辣”“不吃香菜”，应该高于销量、距离和历史偏好。大模型可以负责表达，但核心候选必须由工具和规则控制，尤其在弱模型或低成本模型场景下。

### 补充修复：大模型回复结构化展示

问题：DeepSeek 返回的推荐文案会带 `**`、编号、长段落等 Markdown 风格内容，前端之前用纯文本展示，导致所有内容挤在一起，而且 Markdown 符号直接暴露给用户。

本次修复：

- `prompt.js` 要求大模型的 `improvedReply` 使用中文纯文本分行，不输出 Markdown 符号。
- `app.js` 新增 `renderAgentReply()`，负责把模型回复渲染成有标题、有编号、有分段的版式。
- `app.js` 新增 `cleanAssistantText()`，清理 `**`、反引号、标题符、引用符和列表符号。
- `app.js` 新增 `buildReplyBlocks()`，把文本拆成标题、段落和编号卡片。
- `styles.css` 新增 `.agent-reply`、`.reply-title`、`.reply-item` 等样式。

产品经理理解：

大模型负责“内容生成”，前端负责“信息呈现”。不要依赖模型输出 Markdown 来完成 UI 排版，因为模型输出不稳定。更稳的做法是：模型给结构清晰的文本，前端做统一渲染。

### 补充修复：版式解析器误拆数字

问题：页面把“18 分钟、3980 单、22 分钟”等数字误识别成新的推荐编号，导致一个餐厅被拆成多个卡片。

原因：前端用正则从自然语言里猜结构，遇到业务数字就误判。

修复：

- `server.js` 给推荐结果增加 `displayPayload`，直接返回结构化展示数据。
- `app.js` 新增 `renderDisplayPayload()`，按字段渲染标题、指标、核心餐品、匹配原因。
- 推荐类回复不再依赖自然语言解析。
- 移除“任意数字 + 空格”识别为编号的规则。

产品经理理解：推荐结果本质是结构化信息，不应该让前端从自然语言里反推结构。行业最佳实践是“模型生成内容 + 工具返回结构化数据 + 前端按结构渲染”。

### 补充修复：短期记忆与商品选择

问题：用户前面已经表达了预算、配送时间和清淡诉求，也已经选了餐厅和商品，但输入“低脂牛肉谷物碗”后，系统又追问预算和配送时间。

原因：系统只有长期用户画像和上一轮餐厅选择，没有把“上一轮推荐了哪些商品、用户现在是在选择商品”作为短期会话状态保存。

修复：

- `appState` 新增 `selectedRestaurant` 和 `selectedDish`。
- 前端保留上一轮 `dishRecommendations`。
- 用户输入商品名或“第一个/第二个/第三个”时，先匹配上一轮商品推荐。
- 命中后直接进入 `dish_selection`，不再重新跑点餐意图识别和缺槽追问。
- 右侧状态面板新增 `ShortTermSelection`，展示当前本轮选择。

产品经理理解：

Agent 的 Memory 分两类：

- 长期记忆：用户稳定偏好，例如不吃香菜、喜欢川菜。
- 短期记忆：本轮对话上下文，例如刚推荐了哪 3 家店、用户选了哪家、当前 5 个商品是什么、用户选了哪个商品。

没有短期记忆，Agent 就会像每句话都是第一次听见一样，导致重复追问和上下文断裂。

### 补充修复：大模型失败时的产品化兜底回复

你问“你是什么模型”时，页面返回了“我已经根据你是什么模型完成规则版处理……补充预算、配送时间或口味偏好”。

这个回复是错误的产品体验。原因是 OpenAI API 返回了 `insufficient_quota`，服务端没有拿到真实模型回复，于是展示了过于工程化的 fallback 文案。

本次修复：

- `buildFallbackReview()` 增加模型身份问题识别。
- 即使真实大模型调用失败，用户问“你是什么模型”时也会回答当前配置的 provider/model。
- `server.js` 增加 `buildModelErrorReview()`。
- 当 OpenAI 返回额度不足时，回复会明确说明“当前配置是 OpenAI 的模型，但 API 额度不足或账单未开通”。
- `app.js` 的 `LLMState` 会展示接口错误原因，方便判断是模型输出差，还是接口没成功。

产品经理理解：

Agent 的失败状态也要被产品化。不能把“规则版处理”“Skills”“Workflow 状态”这种内部工程语言直接丢给用户。用户需要知道的是：系统现在能做什么、哪里失败了、下一步该怎么处理。

### 这次为什么要改

DeepSeek 当前输出质量不稳定，会影响你判断 Agent 架构本身是否有效。

这一步把大模型供应商切换到 OpenAI，让 Agent 中枢使用 GPT 模型生成最终回复。更重要的是，我们把“模型身份”从写死文本改成读取服务端配置，避免以后换模型时回答错误。

### 产品更新

- `.env.example` 默认改为 OpenAI 配置。
- `.env` 已切换为 `LLM_PROVIDER=openai`。
- `LLM_MODEL` 默认改为 `gpt-5.5`。
- `LLM_BASE_URL` 改为 `https://api.openai.com/v1`。
- `prompt.js` 删除写死的 DeepSeek 身份说明。
- `server.js` 会把当前 `provider` 和 `model` 传给 Prompt。
- 出于安全考虑，没有把已经暴露在聊天里的 OpenAI API Key 写入本地文件。

### 文件和函数目的

#### `.env`

核心意图：保存本地运行时配置。

用于做什么：告诉服务端当前使用哪家模型供应商、哪个模型、哪个 API 地址和哪个密钥。

产品经理理解：`.env` 是“环境配置”，不是业务代码。不同环境可以使用不同模型，例如本地测试用 GPT，低成本测试用 mini 模型，生产环境用更稳定的配置。

#### `server.js` 的 `modelRuntime`

核心意图：把服务端真实模型配置传给大模型 Prompt。

用于做什么：让大模型知道自己当前是 OpenAI 还是 DeepSeek、具体模型名是什么。

产品经理理解：Agent 的身份和能力边界应该来自真实运行状态，而不是写死在 Prompt 里。

#### `prompt.js` 的 `formatProviderName()`

核心意图：把机器配置里的供应商名称转成用户能理解的名称。

用于做什么：用户问“你是什么模型”时，可以回答“当前系统接入了 OpenAI 的 gpt-5.5”。

产品经理理解：这是一个小函数，但它解决的是“用户信任”和“系统可解释性”的问题。

### 安全提醒

你刚才发出的 API Key 已经暴露在聊天上下文里。最佳实践是：

1. 到 OpenAI Platform 撤销这个 Key。
2. 重新生成一个新的 Key。
3. 只把新 Key 填到本地 `.env` 的 `OPENAI_API_KEY=` 后面。
4. 不要再把 Key 发到聊天、截图、文档或前端代码里。

### 核心原理

S01 让我们理解 Agent 是一个循环。S02 让我们把能力拆成工具。Step 4 则把工具按业务顺序组织起来。

这一步也借鉴了 S05 的计划思想：复杂任务不要直接做，而是先拆成步骤，再一步步推进。

外卖点餐的固定 Workflow 是：

1. 解析需求
2. 判断是否需要追问
3. 读取长期记忆
4. 搜索候选餐厅
5. 查询菜单并排序菜品
6. 生成购物车草稿
7. 等待用户确认

### 技术实现解释

Step 3 中，`app.js` 直接调用工具。Step 4 中，`app.js` 不再关心业务顺序，而是调用：

```js
orderingWorkflow.run(text)
```

Workflow 返回：

- `status`
- `need`
- `memories`
- `recommendations`
- `cart`
- `toolCalls`
- `workflowState`
- `reply`

页面只负责展示这些结果。

### 产品经理理解

Workflow 和 Agent 不是一回事：

- Workflow：路径固定，稳定、可测、适合一期产品
- Agent：路径动态，灵活，但更难控制

你的外卖点餐助手一期应该先做 Workflow，因为点餐链路有明确业务顺序。等 Workflow 稳定后，再对复杂需求加入 ReAct 式动态规划。

### 核心原理

Step 5 在 Agent Loop 前增加了一层 Router：

```text
用户输入 -> IntentResult -> Route -> Tool / Workflow / RAG / Planning / Memory
```

这让系统不再把所有输入都塞进点餐 Workflow，而是根据意图分发。

### 技术实现解释

当前是规则版意图识别：

- 用关键词和槽位抽取识别意图
- 用简单打分选择最高分意图
- 用 `requiredSlots` 判断缺槽
- 用 `route` 决定能力分发

后续接真实大模型后，可以让模型输出同样的结构：

```json
{
  "intent": "order_recommendation",
  "route": "workflow",
  "slots": {
    "budget": 35,
    "maxDeliveryMinutes": 20,
    "tasteGoals": ["清淡"]
  }
}
```

这样前端、Workflow 和工具层都不用推倒重来。

### 产品经理理解

意图识别是 Agent 产品的入口分流系统。它决定用户这句话应该去哪里：

- 问知识 -> RAG
- 找餐厅 -> 单工具
- 点外卖 -> Workflow
- 多人复杂点餐 -> Planning
- 改偏好 -> Memory
- 信息不够 -> Clarify

这一步做扎实，后续接大模型、RAG、高德地图 API 都会更顺。

### 核心原理

RAG 的基本链路是：

```text
用户问题 -> 检索知识库 -> 得到相关片段 -> 基于片段回答
```

在 Agent 里，RAG 通常作为一个工具存在：

```text
Intent route = rag_lookup
-> call retrieve_food_knowledge
-> return KnowledgeResults
-> Agent / Workflow 生成回答
```

### 技术实现解释

当前是关键词版 RAG：

- 知识库存在 `knowledge-base.json`
- `rag.js` 根据 query 和 slots 生成检索词
- 命中标签、标题、摘要会加分
- 取 topK 知识片段
- 用模板生成回答

这不是最终生产形态，但它让你完整理解 RAG 的骨架。

### 产品经理理解

外卖 Agent 需要两类信息：

- 结构化业务数据：餐厅、菜单、价格、配送
- 非结构化知识：什么适合暖胃、什么适合减脂、哪些菜系偏辣、过敏要注意什么

RAG 解决的是第二类问题。

## Step 19：LLM 意图识别与规则校验

### 本步解决的问题

之前的意图识别主要靠 `intent.js` 里的规则完成。规则的好处是稳定、可测试、不会乱编；但缺点是面对真实用户的口语表达、多轮改口、隐含条件和复杂上下文时，很容易漏掉。

所以这一步把意图识别升级成混合架构：

```text
用户输入
-> 短期上下文继承
-> LLM 意图识别和槽位抽取
-> 规则校验与兜底
-> Workflow / RAG / Tool / Planning / Memory
-> 工具结构化结果
-> LLM 最终回复
```

### 为什么不完全交给大模型

大模型擅长理解自然语言，但不适合单独承担全部控制权。

如果完全靠大模型，可能出现：

- 输出 JSON 不稳定
- 漏掉预算、配送时间、忌口等关键槽位
- 把“你是什么模型”误路由到知识库或推荐链路
- 把支付、下单、过敏、长期记忆这类高风险边界处理错
- 自动化评测结果不稳定

所以最佳实践不是“规则或 LLM 二选一”，而是让它们分工：

- LLM：理解用户说法，抽取意图和槽位
- 规则：校验字段、守住边界、失败兜底
- Workflow：按结构化任务单执行流程
- Tools：只处理确定输入，不直接理解自然语言
- 最终 LLM：整合工具结果并回复用户

- 新增 `llm-intent.js`

- 页面右侧新增 `LLMIntentState`
- `package.json` 的 `npm run check` 覆盖新链路

### 新增文件和函数的目的

`llm-intent.js`

核心目的：把大模型输出变成可靠的、机器可执行的意图任务单。

- `LLM_INTENT_SCHEMA`：定义模型应该返回哪些字段
- `buildIntentParsePrompt()`：指导模型如何识别意图、槽位、冲突和上下文继承
- `normalizeLLMIntentParse()`：把模型结果补齐成 Workflow 能用的 `IntentResult`
- `buildFallbackIntentParse()`：模型不可用时退回规则版识别
- `safeParseIntentText()`：安全解析模型 JSON 输出

`/api/intent-parse`

核心目的：通过服务端代理调用真实大模型，避免浏览器暴露 API Key。

## Step 20：Agent 稳定性重构

### 本步为什么必须做

用户真实使用 Agent 时，问题往往不是“完全不会回答”，而是“一句话就暴露明显场景漏洞”。

这说明系统缺的不是更多 prompt，而是稳定性底座：

- 多轮对话状态：知道用户这一句是在改口、继续、选店、换一批，还是问模型
- 约束引擎：把“20 分钟以内”“不要刚才三家”“清淡一点”“不吃牛肉”变成推荐前必须满足的条件
- 曝光历史：记住用户看过哪些餐厅，避免换一批时重复推荐
- 自动化评测：把真实生活场景变成每次提交前都能跑的检查

### 新增文件的目的

`dialogue-state.js`

核心目的：管理短期对话状态。

它会识别本轮对话动作，例如：

- `new_order_request`：新的点餐需求
- `refine_constraints`：修改上一轮条件
- `request_alternatives`：想看上一批之外的餐厅
- `select_restaurant`：用户选第几家店
- `select_dish`：用户选第几个商品
- `ask_agent_identity`：询问模型或能力边界

`constraint-engine.js`

核心目的：在推荐结果展示前做硬约束校验。

它把用户需求拆成两类：

- 硬约束：严格配送时间、忌口、排除餐厅、清淡不能出重口、重口不能出纯轻食
- 软偏好：预算、销量、距离、长期记忆、用餐场景

这个区分非常重要。硬约束不能违反；软偏好用于排序。

`evals/stability-matrix.js`

核心目的：把最容易出错的真实场景固定成自动化回归测试。

覆盖场景包括：

- 清淡改重口
- 严格 20 分钟
- 换一批并排除上一批
- 问模型后继续刚才需求
- 选店后换其他餐厅
- 不吃牛肉、不要辣、高蛋白

### 这一步修正的关键原则

长期记忆不能覆盖本轮 query。

之前的问题是：当用户没有明确说口味时，系统会把长期画像里的偏好写进 `tasteGoals`，这会把软偏好错误升级成硬约束。

正确做法是：

```text
本轮明确 query > 多轮短期状态 > 安全和权限边界 > 长期记忆软排序 > 默认假设
```

也就是说，长期记忆可以影响排序，但不能替用户创造本轮约束。

2. 多人强冲突场景还需要更细的拆分策略。
