# 20 个真实多轮场景体验验收

评测日期：2026-07-10

评测对象：本地 Web 产品 `http://127.0.0.1:5177/`

模型链路：DeepSeek `deepseek-v4-flash`，真实调用 `/api/intent-parse` 和 `/api/llm-review`

## 评测目的

这次评测不只判断代码能否运行，而是以真实用户完成点餐任务为标准，检查：

- 用户意图是否理解正确
- 多轮改口后是否保留未被推翻的条件
- 硬约束是否始终被满足
- 换一批时是否避免重复推荐
- 多人需求是否按参与者拆分
- RAG、Memory、Planning 和点餐主流程能否连续协作
- 最终回复是否只基于工具事实
- Mock 数据边界是否向用户诚实说明
- 沟通是否高效

## 总结论

本轮共执行 20 个场景、39 轮真实页面交互。

- 通过：5 个
- 部分通过：4 个
- 失败：11 个
- 平均单轮耗时：16.5 秒
- 中位耗时：14.5 秒
- P95 耗时：26.6 秒
- 最大耗时：34.6 秒
- 最终回复模型未产出可解析 JSON、转入兜底：37 / 39 轮

当前产品可以演示单人、单目标、约束清晰的点餐流程，但还不适合作为稳定 Demo 对外体验。主要风险不是页面或 Mock 数据量，而是多轮状态策略、多人任务模型、换批语义、事实边界和最终回复来源不一致。

## P0 问题：修复前不建议继续扩功能

### 1. 非点餐轮次会覆盖点餐状态

身份问答、知识问答等旁路任务完成后，`appState.userNeed` 仍被新 Workflow 结果覆盖，导致原点餐条件永久丢失。

正确原则：

```text
旁路问答可以产生临时结果，但不能修改 activeOrderTask。
只有 new_order_request 或 refine_constraints 才能提交新的点餐状态。
```

### 2. 多人需求没有参与者级约束模型

当前只有一个全局 `UserNeed`。两个人口味相反时，系统只能覆盖或混合条件，不能表达：

```text
participant A: 想吃辣
participant B: 绝对不辣
shared: 总预算 80，35 分钟
```

需要新增 `ParticipantNeed[]`，再做共同餐厅可行性和分菜规划。

### 3. 换批识别依赖少量正则表达式

“换一家”可以命中，但“换家店”“这几家不想吃”“还有完全不同的吗”会漏判。漏判后，曝光历史虽然存在，却没有进入排除条件。

应改为：LLM 输出 `dialogueAct`，规则只校验；同时增加语义回归集，不再把正则作为主要理解器。

### 4. 最终回复可能与工具结果矛盾

C04 中工具状态是 `no_match`、菜品列表为空，回复却列出 4 道菜。这违反“工具结果是事实源”的基本原则。

正确原则：

```text
LLM 可以解释、总结和追问，但不能创造不在 ToolResult 中的餐厅、商品或指标。
```

### 5. 数据真实性边界没有强制门禁

当用户要求“真实附近、实时配送、不要模拟数据”时，系统必须先检查数据源能力。如果当前只有 Mock，应直接说明暂不具备，而不是继续展示模拟结果。

## P1 问题：影响完成率和信任

### 6. 单句多意图只能保留一个主路由

“过敏信息 + 今天点餐”被路由成记忆写入，当前任务被阻塞。应把意图解析结果升级为：

```text
primaryTask: 当餐推荐
sideEffects: [待确认长期记忆]
```

### 7. 配送硬约束判定不符合用户语言

“20 分钟内”通常已经是上限约束，不应默认允许 28 分钟。只有“20 分钟左右、最好 20 分钟”才是软偏好。

同时，当前规则会因为预算表达“25 块以内”而把配送时间标为严格，说明严格性判断没有绑定到具体槽位。

### 8. 知识问答与点餐关键词互相污染

“高蛋白晚餐是不是容易胖”因为包含“晚餐”，被判成点餐请求。意图分类应先判断句子的言语行为，而不是只要出现餐次词就算点餐。

### 9. 追问策略没有按信息价值排序

已经有预算、时间、口味时仍追问午餐还是晚餐；五人 100 元时却没有先确认总预算还是人均。

应采用：只追问会改变可执行结果的最高价值槽位，每轮最多一个问题。

## P2 问题：体验和可观察性

### 10. 最终 LLM 成功状态不真实

39 轮中有 37 轮的 DeepSeek 最终回复未解析成功，系统使用兜底；但 `LLMState.mode` 仍显示 `real`。推荐卡又会被服务端结构化渲染覆盖，因此用户很难判断最终内容来自模型还是规则。

建议拆分状态：

```text
intentSource: llm | rules_fallback
replySource: llm | structured_renderer | fallback
parseStatus: success | repaired | failed
```

### 11. 两次串行模型调用导致响应过慢

当前每轮先做意图模型调用，再做最终回复模型调用，平均 16.5 秒，P95 26.6 秒。模糊需求最高 34.6 秒。

短期可以：

- 餐厅和商品卡由结构化渲染直接展示，不等待第二次 LLM
- 只有知识问答、冲突解释、复杂规划才调用最终回复模型
- 展示阶段状态，例如“正在理解需求”“正在筛选餐厅”，而不是长时间只有“正在思考”

## Mock 数据需要补充的部分

Mock 数据不是本轮多数错误的主因，但以下字段会限制下一阶段：

- 地址或服务区域绑定，不能再让所有餐厅同时属于中关村和常营
- 营业时间、夜宵可用性、库存状态
- 餐厅人均价、起送价、配送费，支持餐厅级预算筛选
- 菜品可定制项，例如不辣、少油、去香菜
- 过敏原交叉污染声明
- 儿童适配、多人套餐、荤素组合、份量和可分享人数
- 数据来源标记：`mock`、`merchant`、`map_estimate`、`realtime`

## 推荐修复顺序

1. 建立 `ConversationState` 与 `ActiveOrderTask` 的提交规则，禁止旁路问答覆盖点餐状态。
2. 把 20 个场景固化为可自动运行的 LLM 集成评测，并记录每轮状态快照。
3. 修复换批语义和已展示餐厅排除逻辑。
4. 建立事实门禁，保证最终回复只能引用工具结果，并标明数据来源。
5. 实现 `primaryTask + sideEffects` 的多意图编排。
6. 实现 `ParticipantNeed[]` 和多人共同约束规划。
7. 重做槽位级硬/软约束判定与追问优先级。
8. 优化两次模型调用和回复来源可观察性。
9. 完成以上回归后，再接高德地图 API。

## 本轮验收标准变化

以前的自动化评测主要运行规则 Workflow，因此即使通过，也不能证明真实页面和真实 LLM 多轮体验可靠。

从现在开始，验收分成三层：

1. 单元层：意图、约束、排序函数是否正确。
2. 集成层：Dialogue、LLM Intent、Workflow、Tool、Memory 是否正确传递状态。
3. 体验层：通过真实网页和真实模型执行多轮用户故事，检查最终可见结果。

只有三层全部通过，功能才可以在进度中标记为“验收通过”。

## 修复后 bad case 回归（2026-07-10）

### 修复结论

原体验评测暴露的 P0/P1 机制问题已经完成修复，不再以“补单个 query”作为解决方式，而是增加会话提交策略、事实门禁、多意图副作用和参与者级规划。

### 自动化回归

- `npm run eval:bad`：20/20 通过
- `npm run eval`：6/6 通过
- `npm run eval:stability`：6/6 通过
- `npm run eval:free`：20/20 通过
- `npm run eval:complex`：17/17 通过
- `npm run check`：全部语法检查通过

20 个 bad case 覆盖：

- 清淡改重口后仍保留配送硬约束
- 连续两次换批不重复餐厅
- 选店后新增忌口、选菜后反悔换店
- 身份/知识插话不覆盖当前点餐任务
- 过敏记忆和当餐推荐同时执行
- 两人相反辣度、家庭三人不同约束
- 预算硬约束与配送软约束不串槽
- 五人预算总额/人均歧义追问
- 真实数据能力边界
- 严格素食、过敏原安全展示
- “35 分钟左右都行”不再误判成“吃什么都行”
- “胃不舒服，改清淡少油”识别为当前订单改口，而不是知识问答
- “过敏 + 指定餐厅菜单”同时查询安全菜品并生成长期记忆确认
- 多人预算从“合计”纠正为“每个人”后，预算范围、总预算和餐厅排序同步更新

### 真实浏览器 + 真实模型复测

已通过页面和 DeepSeek 重跑以下高风险链路：

1. 一家三口两轮对话：稳定保持 3 人，分别保留“我吃辣、老婆避羊肉、孩子不辣少油”。
2. 身份问题插入：回答当前模型后，原预算、时间和口味仍可继续换批。
3. 饮食知识插入：回答高蛋白知识后，恢复原清淡、高蛋白、预算和配送条件。
4. 预算更新：`25 块以内`只把预算改为硬约束，不会把配送时间误改成硬约束。
5. 团队预算：5 人 100 元优先追问总预算还是人均。
6. 真实数据请求：当前仅有 Mock 时直接阻断，不展示模拟附近餐厅。
7. 双重过敏：继续给出安全餐厅，同时生成高风险长期记忆确认；卡片核心餐品也经过过敏原过滤。

### 当前结论

本轮发现的机制级问题已解决并有回归保护。产品可以继续进入高德地图和真实商家数据接入阶段，但不能把当前 Mock 距离和配送时间解释为真实外卖平台数据。
